1. 🎯 Tổng Quan Hệ Thống

Giới Thiệu Dự Án

VendingAd System là một CMS (Content Management System) để quản lý lịch phát video trên màn hình máy bán hàng tự động và TV box. Hệ thống cho phép người dùng upload video, tạo playlist, và lên lịch phát nội dung cho nhiều thiết bị.

Quy mô triển khai dự kiến:

  • 5-10 người dùng (user)
  • Mỗi user có thể quản lý 50+ thiết bị
  • Tổng cộng khoảng 200-500 thiết bị đồng thời
  • Thiết bị poll API mỗi 30-60 giây

2. 🏗️ Kiến Trúc & Thiết Kế

2.1. Kiến Trúc Tổng Thể

✅ Điểm Mạnh

  • Clean Architecture: Phân tách rõ ràng Domain, Application, Infrastructure
  • Event-Driven Architecture: Đã tích hợp RabbitMQ để xử lý cache invalidation bất đồng bộ
  • CQRS Pattern (nhẹ): Tách biệt read queries (AsNoTracking) và write operations
  • Repository Pattern: Abstraction layer cho data access
  • Dependency Injection: Sử dụng DI container của .NET đúng cách
  • Multi-database Support: Hỗ trợ SQLite, PostgreSQL, SQL Server thông qua configuration

2.2. Domain Model

Entities chính:

  • User - Người dùng CMS
  • Admin - Quản trị viên
  • Device - Thiết bị TV box/vending machine
  • Media - Video đã upload
  • Playlist - Template danh sách phát
  • PlaylistItem - Item trong playlist
  • PlaybackSchedule - Lịch phát thực tế
  • PlaybackScheduleDevice - Liên kết schedule-device (many-to-many)
  • PlaybackScheduleItem - Snapshot media trong schedule

✅ Domain Model Tốt

  • Phân biệt rõ Playlist (template) vs PlaybackSchedule (actual plan)
  • Schedule items là snapshot, không bị ảnh hưởng khi playlist thay đổi
  • Hỗ trợ một schedule áp dụng cho nhiều devices
  • Device có claim mechanism để provisioning an toàn

2.3. Database Design

✅ Indexing Strategy Tốt

  • Unique index trên Device.DeviceCode
  • Composite index cho active schedule queries: (IsActive, StartDate, EndDate)
  • Index cho user-scoped queries: (UserId, IsActive, CreatedAt)
  • Foreign key indexes cho join performance

⚠️ Vấn Đề Tiềm Ẩn

  • Cascade Delete: Một số relationship dùng DeleteBehavior.Cascade có thể gây xóa dữ liệu không mong muốn. Ví dụ: xóa User sẽ cascade xóa tất cả Playlists và Schedules của user đó.
  • Thiếu Soft Delete: Không có cơ chế soft delete cho audit trail. Khi xóa schedule/device, mất hết lịch sử.
  • Thiếu Audit Table: Không có bảng ghi lại ai làm gì, khi nào (login, create schedule, delete video, rotate device secret, etc.)

2.4. Caching Strategy

✅ Multi-Layer Cache Hiệu Quả

  • Per-device cache: mobile:playback-state:{deviceCode} - giảm DB load cho polling
  • Shared schedule cache: mobile:schedule-content:{scheduleId}:{version} - nhiều device dùng chung
  • Device presence cache: device:online:{deviceCode} với TTL - giảm DB writes từ heartbeat
  • Distributed lock: lock:mobile:schedule-content:* - tránh cache stampede
  • Event-driven invalidation: Worker invalidate cache khi schedule thay đổi, không block web request

💡 Cải Thiện Cache

  • Thêm cache metrics: hit rate, miss rate, eviction count
  • Implement cache warming strategy khi deploy mới
  • Xem xét thêm cache cho user media list (ít thay đổi, query nhiều)
  • TTL hiện tại chưa được document rõ ràng trong code

3. 💎 Chất Lượng Code

3.1. Code Organization

✅ Tổ Chức Tốt

  • Naming conventions nhất quán (PascalCase cho class/method, camelCase cho parameters)
  • File structure theo namespace rõ ràng
  • Service interfaces và implementations tách biệt
  • DTOs tách riêng khỏi entities
  • Nullable reference types enabled (<Nullable>enable</Nullable>)

3.2. SOLID Principles

✅ Tuân Thủ SOLID

  • Single Responsibility: Mỗi service có trách nhiệm rõ ràng (MediaService, DeviceService, PlaybackScheduleService)
  • Open/Closed: Strategy pattern cho database providers, cache implementations
  • Liskov Substitution: ICacheService có thể swap giữa Redis và MemoryCache
  • Interface Segregation: Interfaces nhỏ, focused (ITimeService, IPasswordHashingService)
  • Dependency Inversion: Controllers phụ thuộc vào interfaces, không phụ thuộc concrete implementations

3.3. Error Handling

⚠️ Error Handling Chưa Đồng Nhất

  • Một số service throw exceptions, một số return result objects
  • Không có global exception handler middleware
  • Error messages trả về client đôi khi expose internal details
  • Thiếu structured logging cho errors
  • RabbitMQ publish failures chỉ log warning, không có retry mechanism

💡 Đề Xuất

  • Implement Result<T> pattern hoặc OneOf<Success, Error> để tránh exceptions cho business logic failures
  • Thêm global exception handler middleware để catch unhandled exceptions
  • Standardize error response format (ProblemDetails RFC 7807)
  • Thêm correlation ID cho request tracking

3.4. Async/Await Usage

✅ Async Pattern Đúng

  • Tất cả DB operations đều async
  • Không có blocking calls (.Result, .Wait())
  • ConfigureAwait không cần thiết trong ASP.NET Core (đúng)

4. 🔒 Bảo Mật

4.1. Authentication & Authorization

✅ Đã Làm Tốt

  • Cookie Authentication: HttpOnly, SameSite=Lax, 24h expiration với sliding
  • Role-based Authorization: Admin vs User roles
  • Device Authentication: Device secret với bcrypt hashing
  • Password Hashing: Sử dụng ASP.NET Core Identity PasswordHasher (PBKDF2)
  • Anti-forgery Tokens: Enabled cho POST requests

HIGH Vấn Đề Bảo Mật Cần Xử Lý

1. Mật khẩu mặc định cố định (TD@12345)

  • Vị trí: AdminController.cs - CreateUser, ResetPassword
  • Rủi ro: Tất cả user mới đều có password giống nhau. Nếu một user quên đổi password, attacker có thể đăng nhập.
  • Ảnh hưởng: Unauthorized access vào CMS, có thể xem/sửa/xóa schedules và devices của user khác
  • Khuyến nghị:
    • Generate random password khi tạo user mới
    • Hiển thị password một lần duy nhất cho admin
    • Thêm flag MustChangePassword
    • Force user đổi password sau lần login đầu tiên

2. Thiếu rate limiting cho login endpoint

  • Vị trí: AccountController.cs - Login action
  • Rủi ro: Brute force attack có thể thử nhiều password
  • Khuyến nghị: Thêm rate limit cho login attempts (5 attempts / 15 minutes per IP hoặc username)

3. Session management chưa có timeout tuyệt đối

  • Hiện tại: Sliding expiration 24h, session có thể kéo dài vô hạn nếu user active liên tục
  • Khuyến nghị: Thêm absolute timeout (ví dụ: 7 days) để force re-authentication

4. Thiếu audit log cho sensitive operations

  • Vị trí: Login, logout, create/delete user, rotate device secret, upload/delete video
  • Rủi ro: Không thể trace back khi có incident
  • Khuyến nghị: Tạo bảng AuditLogs ghi lại actor, action, target, timestamp, IP

4.2. Input Validation

✅ Upload Validation Tốt

  • File extension whitelist (.mp4, .mov, .webm)
  • MIME type validation
  • Magic bytes check (file signature)
  • ffprobe validation cho video codec và duration
  • File size limit (50MB)
  • Filename sanitization với Path.GetFileName

MEDIUM Input Validation Gaps

  • Một số controller actions thiếu model validation
  • SQL injection risk thấp (dùng EF Core parameterized queries) nhưng chưa có explicit validation cho special characters
  • XSS risk thấp (Razor auto-encode) nhưng nên thêm Content Security Policy headers

4.3. Secrets Management

✅ Cải Thiện Đáng Kể

  • Seed:EnableDemoData=false mặc định trong appsettings.json
  • RabbitMQ credentials không còn hardcode trong appsettings.json
  • Startup fail nếu bật demo seed ngoài Development environment
  • Device secrets được hash với bcrypt, không lưu plaintext

MEDIUM Secrets Management Chưa Hoàn Chỉnh

  • appsettings.Development.json: Có thể chứa secrets thật, cần thêm vào .gitignore
  • SQL Server password: Trong docker-compose.infra.yml là VendingAd@12345 (chỉ dùng local, nhưng nên document rõ)
  • Production secrets: Chưa có hướng dẫn rõ ràng về cách inject secrets qua environment variables hoặc secret manager
  • Khuyến nghị:
    • Dùng User Secrets cho local development
    • Dùng Azure Key Vault / AWS Secrets Manager / HashiCorp Vault cho production
    • Document secret rotation process

4.4. API Security

✅ Mobile API Security

  • Device secret required cho tất cả mobile endpoints
  • Support cả X-Device-Secret header và Authorization: Bearer
  • Device secret có rotate và revoke mechanism
  • Rate limiting per device code (không chỉ per IP)

💡 Cải Thiện API Security

  • Thêm HTTPS enforcement trong production (HSTS headers)
  • Implement API versioning (/api/v1/mobile/...)
  • Thêm request signing để verify request integrity
  • Consider JWT tokens thay vì static device secrets cho better rotation

5. ⚡ Hiệu Năng & Khả Năng Mở Rộng

5.1. Database Performance

✅ Query Optimization Tốt

  • AsNoTracking() cho tất cả read-only queries
  • Eager loading với Include() để tránh N+1 queries
  • Composite indexes cho common query patterns
  • Connection pooling mặc định của EF Core

LOW Potential Performance Issues

  • Large result sets: Một số queries không có pagination (GetAllAsync methods)
  • File storage: Video files lưu trên local filesystem, không scale khi có nhiều instances
  • Lack of query timeout: Không có explicit timeout cho DB queries

5.2. Caching Performance

✅ Caching Strategy Xuất Sắc

  • Hit rate cao: Shared schedule cache cho nhiều devices
  • Cache stampede prevention: Distributed lock với Redis
  • Heartbeat throttling: Chỉ write DB mỗi 60s thay vì mỗi request
  • Event-driven invalidation: Cache invalidation không block web requests

5.3. Rate Limiting

✅ Rate Limiting Implementation

  • Device-based rate limiting (không chỉ IP-based)
  • Separate limits cho từng endpoint type:
    • Heartbeat: 10 requests/60s
    • Playback state: 30 requests/60s
    • Device info: 20 requests/60s
  • HTTP 429 response với Retry-After header

MEDIUM Rate Limiting Limitations

  • In-memory only: Rate limiter hiện tại là in-process, không work với multiple instances
  • Khuyến nghị: Khi scale horizontal, cần move rate limit counters sang Redis

5.4. Scalability

💡 Scalability Considerations

  • Horizontal scaling: Web app có thể scale horizontal (stateless), nhưng cần:
    • Shared file storage (S3, Azure Blob, Cloudflare R2)
    • Redis-based rate limiting
    • Session store trong Redis (hiện tại dùng in-memory)
  • Worker scaling: Worker có thể scale horizontal (RabbitMQ queue-based)
  • Database scaling:
    • Read replicas cho read-heavy queries
    • Connection pooling đã có
    • Cần monitor slow queries

5.5. Load Testing

MEDIUM Chưa Có Load Testing

  • Chưa verify hệ thống handle được 200-500 devices polling đồng thời
  • Chưa biết p95/p99 latency trong production-like load
  • Khuyến nghị: Implement k6 hoặc NBomber tests cho:
    • Heartbeat scenario (500 devices, mỗi 30s)
    • Playback state polling (500 devices, mỗi 60s)
    • Schedule update fan-out (1 schedule update → 100 devices invalidate cache)

6. 🛡️ Độ Tin Cậy & Error Handling

6.1. Error Handling

MEDIUM Error Handling Inconsistent

  • Một số services throw exceptions, một số return result objects
  • Không có global exception handler middleware
  • RabbitMQ publish failures chỉ log warning, không retry
  • Worker handler failures acknowledge message (không requeue), dựa vào TTL fallback

💡 Cải Thiện Error Handling

  • Thêm global exception handler middleware
  • Implement retry policy cho transient failures (Polly library)
  • Thêm circuit breaker cho external dependencies (Redis, RabbitMQ)
  • Implement dead letter queue (DLQ) cho failed messages
  • Structured error logging với correlation IDs

6.2. Data Consistency

✅ Transactional Consistency

  • DB changes commit trước khi publish events (at-least-once delivery)
  • Schedule items là snapshot, không bị ảnh hưởng bởi playlist changes
  • Device assignment changes publish union của old + new device codes

LOW Eventual Consistency Gaps

  • Nếu RabbitMQ publish fail, cache không được invalidate (dựa vào TTL)
  • Nếu Worker crash giữa chừng, một số device caches không được invalidate
  • Khuyến nghị: Implement transactional outbox pattern khi message reliability trở nên critical

6.3. Health Checks

✅ Health Checks Comprehensive

  • /health/live - process running check
  • /health/ready - database, Redis, RabbitMQ connectivity
  • Worker validates dependencies on startup
  • JSON response format cho automation

6.4. Graceful Degradation

✅ Fallback Mechanisms

  • Redis disabled → fallback to MemoryCache
  • RabbitMQ disabled → NullMessagePublisher (no-op)
  • ffprobe not available → skip validation (configurable)
  • Cache miss → query database

MEDIUM 6. Thiếu Global Exception Handler

Vấn đề: Unhandled exceptions expose stack traces

Fix: Thêm exception handler middleware

app.UseExceptionHandler(errorApp =>
{
    errorApp.Run(async context =>
    {
        var exceptionHandlerFeature = context.Features.Get<IExceptionHandlerFeature>();
        var exception = exceptionHandlerFeature?.Error;

        _logger.LogError(exception, "Unhandled exception");

        context.Response.StatusCode = 500;
        context.Response.ContentType = "application/json";

        await context.Response.WriteAsJsonAsync(new
        {
            success = false,
            message = "Đã xảy ra lỗi. Vui lòng thử lại sau.",
            correlationId = context.TraceIdentifier
        });

12.2. Không Nên Làm (Anti-patterns Cần Tránh)

  • Không auto-apply migrations on startup trong production: Risk của concurrent migrations
  • Không log passwords/secrets/tokens: Cần policy rõ ràng
  • Không dùng .Result hoặc .Wait(): Deadlock risk (hiện tại đã tốt)
  • Không cascade delete User: Sẽ mất hết schedules/playlists, nên soft delete
  • Không expose stack traces ra client: Security risk

12.3. Technical Debt Cần Xử Lý

  • Error handling inconsistency: Một số throw exceptions, một số return results
  • Lack of correlation IDs: Khó trace requests
  • No retry policies: Transient failures không được handle
  • No circuit breakers: Cascading failures risk
  • No pagination: GetAllAsync methods có thể return quá nhiều data

public class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GlobalExceptionHandlerMiddleware> _logger;

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (Exception ex)
        {
            var correlationId = context.Items["CorrelationId"]?.ToString() ?? context.TraceIdentifier;
            _logger.LogError(ex, "Unhandled exception. CorrelationId: {CorrelationId}", correlationId);

            context.Response.StatusCode = 500;
            context.Response.ContentType = "application/json";

            await context.Response.WriteAsJsonAsync(new
            {
                success = false,
                message = "Đã xảy ra lỗi. Vui lòng thử lại sau.",
                correlationId
            });
        }
    }
}

// Program.cs
app.UseMiddleware<GlobalExceptionHandlerMiddleware>();

// Filters/LoginRateLimitAttribute.cs
public class LoginRateLimitAttribute : ActionFilterAttribute
{
    private static readonly ConcurrentDictionary<string, LoginAttempt> _attempts = new();

    public override void OnActionExecuting(ActionExecutingContext context)
    {
        var ip = context.HttpContext.Connection.RemoteIpAddress?.ToString() ?? "unknown";
        var username = context.ActionArguments.Values
            .OfType<LoginRequest>()
            .FirstOrDefault()?.Username ?? "";

        var key = $"{ip}:{username}";
        var attempt = _attempts.GetOrAdd(key, _ => new LoginAttempt());

        if (attempt.Count >= 5 && DateTime.UtcNow < attempt.LockoutUntil)
        {
            context.Result = new ObjectResult(new
            {
                success = false,
                message = $"Quá nhiều lần đăng nhập thất bại. Vui lòng thử lại sau {(attempt.LockoutUntil - DateTime.UtcNow).TotalMinutes:F0} phút."
            }) { StatusCode = 429 };
        }
    }
}

public class PagedResult<T>
{
    public List<T> Items { get; set; }
    public int TotalCount { get; set; }
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
}

// Usage
public async Task<PagedResult<Media>> GetMediaAsync(int userId, int page = 1, int pageSize = 20)
{
    var query = _medias.Query().Where(m => m.UserId == userId);
    var totalCount = await query.CountAsync();
    var items = await query
        .OrderByDescending(m => m.UploadedAt)
        .Skip((page - 1) * pageSize)
        .Take(pageSize)
        .AsNoTracking()
        .ToListAsync();

    return new PagedResult<Media>
    {
        Items = items,
        TotalCount = totalCount,
        Page = page,
        PageSize = pageSize
    };
}

// Install
dotnet add package Polly

// Infrastructure/Resilience/ResiliencePolicies.cs
public static class ResiliencePolicies
{
    public static IAsyncPolicy<T> GetRetryPolicy<T>()
    {
        return Policy<T>
            .Handle<HttpRequestException>()
            .Or<TimeoutException>()
            .WaitAndRetryAsync(
                retryCount: 3,
                sleepDurationProvider: retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt)),
                onRetry: (outcome, timespan, retryCount, context) =>
                {
                    Log.Warning("Retry {RetryCount} after {Delay}s due to {Exception}",
                        retryCount, timespan.TotalSeconds, outcome.Exception?.Message);
                });
    }

    public static IAsyncPolicy GetCircuitBreakerPolicy()
    {
        return Policy
            .Handle<Exception>()
            .CircuitBreakerAsync(
                exceptionsAllowedBeforeBreaking: 5,
                durationOfBreak: TimeSpan.FromSeconds(30),
                onBreak: (exception, duration) =>
                {
                    Log.Error("Circuit breaker opened for {Duration}s", duration.TotalSeconds);
                },
                onReset: () => Log.Information("Circuit breaker reset"));
    }
}

// Use FFmpeg to extract frame
public async Task<string> GenerateThumbnailAsync(string videoPath)
{
    var thumbnailPath = Path.ChangeExtension(videoPath, ".jpg");
    var process = new Process
    {
        StartInfo = new ProcessStartInfo
        {
            FileName = "ffmpeg",
            Arguments = $"-i \"{videoPath}\" -ss 00:00:01 -vframes 1 \"{thumbnailPath}\"",
            RedirectStandardOutput = true,
            UseShellExecute = false
        }
    };
    process.Start();
    await process.WaitForExitAsync();
    return thumbnailPath;
}
